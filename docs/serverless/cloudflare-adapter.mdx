# Cloudflare Worker + Static Assets Adapter

The **Cloudflare Worker adapter** turns a StatikAPI project into a Cloudflare-oriented deployment with:

- a Worker control plane
- Cloudflare **Static Assets** for `/public/...` routes
- a **private** R2 bucket for private routes
- a KV-backed manifest plus runtime usage counters
- authenticated route-path `POST` webhooks for private rebuilds

The current contract is intentionally split:

- **control plane**: Worker endpoints such as `POST /`, route-path `POST`, and `/_manifest`
- **data plane**: public JSON in Static Assets, private JSON in R2

---

## 1) Scaffolding with `create-statikapi`

The easiest way to get started is with the dedicated template:

```bash
pnpm dlx create-statikapi my-worker --template cloudflare-adapter
# or
npx create-statikapi my-worker --template cloudflare-adapter
```

You’ll be prompted for:

- **Project name**
- **Package manager**
- **private R2 bucket binding** and **bucket_name**
- **KV binding** and **KV namespace id**
- **`STATIK_BUILD_TOKEN`**
- **`STATIK_SRC`**
- **`STATIK_USE_INDEX_JSON`**
- **private auth header name/value**
- **Cloudflare account id** and **API token placeholders**
- **worker request / R2 Class A / R2 Class B limits**

The template:

- Writes a ready-to-use **`wrangler.toml`** with Static Assets, private bucket, KV, and runtime env vars.
- Writes a **`statikapi.config.js`** with project-level Cloudflare defaults.
- Writes a **`.dev.vars.example`** file for local secrets and deploy env templates.
- Keeps everything **JavaScript-only**.
- Includes scripts for local dev, build, and deploy.
- Uses a package-manager-neutral `statikapi-cf dev` flow so `pnpm`, `npm`, and `yarn` can all start the same local preview experience.

After scaffolding:

```bash
cd my-worker
pnpm install   # or yarn / npm
pnpm dev       # runs the local Worker dev loop + preview UI
```

> Exact script names may vary slightly, but the template wires dev/build scripts for you.

## 2) Serving model

- Public routes are emitted under `/public/...`.
- Public routes are generated into the project `public/` assets directory during `statikapi-cf build`.
- Asset-matching `/public/...` requests are intended to be served directly by Cloudflare Static Assets.
- The Worker serves private requests from private storage and continues to own `/build` and `/manifest`.
- Private routes stay at their original route paths and require the configured auth header.

## 3) Route-level Cloudflare config

Route modules can override the project default:

```js
export const config = {
  cloudflare: {
    public: true,
    webhook: true,
  },
};
```

- `public`
  - `true` or omitted emits the route as a public Static Asset under `/public/...`
  - `false` writes the route into the private bucket and keeps the original route path
- `webhook`
  - route-level override for whether `/build` is allowed to rebuild that route

## 4) What the adapter does

At a high level, the Cloudflare adapter Worker:

1. **Bundles your route modules into a Worker registry**
2. **Writes public JSON into Static Assets and private JSON into the private bucket**
3. **Writes the private manifest into KV and merges public routes from the embedded build manifest**
4. **Tracks optional runtime limits in KV**
5. **Exposes `/build` and `/manifest`**
6. **Serves private reads from R2 while public `/public/...` requests are handled directly by Static Assets**

## 5) Required bindings and env vars

In **`wrangler.toml`**:

```toml
[assets]
directory = "./public"
binding = "ASSETS"

[[r2_buckets]]
binding = "STATIK_PRIVATE_BUCKET"
bucket_name = "my-statikapi-private"

[[kv_namespaces]]
binding = "STATIK_MANIFEST"
id = "xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
```

These names should match the ones you passed to `create-statikapi` (or update them consistently).

### b) Env vars

Also in `wrangler.toml`:

```toml
[vars]
STATIK_BUILD_TOKEN = "REPLACE_ME_WITH_LONG_RANDOM_STRING"
STATIK_SRC = "src-api"
STATIK_USE_INDEX_JSON = "false"
STATIK_PRIVATE_BUCKET_BINDING = "STATIK_PRIVATE_BUCKET"
STATIK_PRIVATE_AUTH_HEADER_NAME = "x-statik-private-auth"
STATIK_PRIVATE_AUTH_HEADER_VALUE = "REPLACE_ME_PRIVATE_AUTH_VALUE"
STATIK_WORKER_REQUEST_LIMIT = "0"
STATIK_R2_CLASS_A_LIMIT = "0"
STATIK_R2_CLASS_B_LIMIT = "0"
```

- `STATIK_BUILD_TOKEN`
  - required to call `/build`
- `STATIK_PRIVATE_AUTH_HEADER_NAME` / `STATIK_PRIVATE_AUTH_HEADER_VALUE`
  - required to read private routes from the Worker
- `STATIK_WORKER_REQUEST_LIMIT`
  - hard ceiling for total Worker requests
- `STATIK_R2_CLASS_A_LIMIT`
  - hard ceiling for R2 write-class operations performed through the Worker
- `STATIK_R2_CLASS_B_LIMIT`
  - hard ceiling for R2 read-class operations performed through the Worker

## 6) Webhook behavior

The generated project can disable webhook builds globally in `statikapi.config.js`, and routes can disable them locally.

- If webhook is disabled globally:
  - authorized webhook `POST` rebuilds are rejected
- If webhook is enabled globally but disabled for a route:
  - full webhook builds skip that route
  - targeted webhook rebuilds for that route are rejected

## 7) Deploying and using custom domains

### a) Deploy the Worker

From your project:

```bash
pnpm build   # bundles worker + StatikAPI adapter logic
pnpm deploy  # or `wrangler deploy`, depending on the template scripts
```

(Use the exact scripts from your generated `package.json`.)

You can also deploy via:

```bash
wrangler deploy
```

as long as your `wrangler.toml` and entrypoint match the template.

> **Important:** Deploying the Worker updates code/config and public Static Assets together, but route-path webhooks in this version only refresh private Worker-managed outputs. Public data changes still require a rebuild + redeploy/publish flow.

### b) Attach a custom domain to the Worker

In the Cloudflare Dashboard:

1. Go to **Workers & Pages → your Worker**.
2. Add a **Route** like:

   ```text
   api.example.com/*
   ```

3. Ensure `api.example.com` has a DNS record in the same zone.

This gives you URLs such as:

```text
https://api.example.com/build
https://api.example.com/manifest   (if you expose one)
```

### c) Public routes

Public routes are served from the same Worker deployment through Cloudflare Static Assets.

Example URLs:

```text
https://api.example.com/public/users/1
https://api.example.com/public/docs/getting-started
```

### d) CORS for browser clients

If your API is accessed from a frontend (browser) on another domain, configure **CORS**:

- For R2, set **CORS rules** from the Cloudflare Dashboard for the R2 bucket/custom domain.
- Allow only the specific origins that should be able to read your JSON (e.g. `https://app.example.com`).

This keeps your API consumable from your frontend while avoiding wide-open `*` CORS when you don’t need it.

## 8) Using webhook rebuilds

The Worker uses authenticated `POST` requests for rebuilds:

- `POST /` rebuilds all webhook-enabled private outputs
- `POST /users/1` rebuilds that private route when webhook is enabled
- public routes are not updated through webhooks in this version

### a) Rebuild everything

```bash
curl -X POST "https://api.example.com/build" \
  -H "Authorization: Bearer YOUR_STATIK_BUILD_TOKEN"
```

Behavior:

- Rebuilds the private side of the Worker-managed data plane using `STATIK_SRC`.
- Updates the private manifest in **KV**.
- Public Static Assets are generated by `statikapi-cf build` and require a fresh deploy/publish step to change.

### b) Rebuild a single route

You can target a specific route with `?route=`:

```bash
curl -X POST "https://api.example.com/build?route=/users/1" \
  -H "Authorization: Bearer YOUR_STATIK_BUILD_TOKEN"
```

The adapter will:

- Rebuild only that private route if webhook is enabled for it.
- Update the corresponding entry in the private bucket + KV.

> Targeted rebuilds for public routes are rejected, because public routes are emitted as Static Assets during the build step and are not webhook-refreshable in this version.

> The exact `route` value should match the normalized URL path you serve (e.g. `/users/1`, `/docs/a/b`, etc.).

## 9) API token guidance

If you use a Cloudflare API token for deploy automation, keep it least-privilege. The generated scaffolding expects at least:

- Workers Scripts: Edit
- R2 Storage: Edit
- Workers KV Storage: Edit

Store account-scoped deploy variables outside the committed source tree. The scaffold writes placeholders into `.dev.vars.example` so contributors can copy them locally without checking secrets into Git.

- CI/CD pipelines

Just remember to:

- Use `POST`
- Include the **Authorization** header:

```http
Authorization: Bearer YOUR_STATIK_BUILD_TOKEN
```

## 6) Build token & rotation strategy

### a) Choosing a token

- Generate a **long, random** string (64+ characters).
- Store it only in:
  - `wrangler.toml` (or Wrangler secrets)
  - The systems that need to call `/build`.

Example:

```bash
# using openssl
openssl rand -hex 32
```

### b) Rotating the token

To keep things safe:

1. **Generate a new token**.
2. Update it in:
   - Cloudflare (Wrangler secret / `wrangler.toml`)
   - Any external systems (CMS, CI, backends) that call `/build`.
3. **Deploy** the Worker.

During rotation, you can:

- Temporarily accept both old + new tokens (if you extend the Worker logic), or
- Do a short maintenance window where `/build` calls may briefly fail while everything is updated.

Treat `STATIK_BUILD_TOKEN` like an API key:

- Never log it.
- Never expose it in frontend code.
- Keep it in secrets managers and CI vaults.

## 7) Request flow overview

**Read path (data plane)**:

1. Client requests `GET https://cdn.example.com/users/1/index.json` (or the default R2 public URL).
2. Cloudflare’s R2 + CDN returns the JSON object.
3. Optional: CORS rules on R2 allow browser apps on `https://app.example.com` to consume this.

The Worker is **not** involved here.

**Update path (control plane)**:

1. Some system calls `POST https://api.example.com/build?route=/users/1` with the right token.
2. Adapter runs the StatikAPI build just for `/users/1`.
3. New JSON is written to R2; manifest in KV is updated.
4. Next read from the R2 URL sees the updated data (subject to any caching you configure).

## 8) Git integration and automatic deploys

The smoothest way to operate the Cloudflare adapter is with **Git + Cloudflare Worker Git integration**:

### a) Put the project in Git

Initialize a repository:

```bash
git init
git add .
git commit -m "init: statikapi cloudflare adapter"
```

Push it to GitHub/GitLab/Bitbucket.

### b) Connect the repo to Cloudflare as a Worker

In the Cloudflare Dashboard:

1. Go to **Workers & Pages → Create application → Worker → Connect to Git**.
2. Choose your repository.
3. Set the **build command** to:

   ```text
   npm run build
   # or
   pnpm build
   # or
   yarn build
   ```

   (Use the command that your template’s `package.json` defines. The important part is that it runs the bundling step for the Worker.)

4. Set the **output directory / entry** as recommended by the template (usually just the Worker script; Cloudflare Workers don’t use a static output dir like Pages).

Now, whenever you push changes to the configured branch, Cloudflare will:

- Pull the latest code.
- Run `npm run build` (or equivalent).
- Deploy a new Worker version automatically.

> **Note:** This only updates the Worker code and its environment. It does **not** regenerate your JSON in R2 — that still happens via `/build`.

### c) Local dev & manual deploys with Wrangler

You can also work locally:

```bash
# watch and test locally
wrangler dev

# on every change to src-api (or whatever STATIK_SRC is),
# rebuild and redeploy:
npm run build
wrangler deploy
```

(Replace `npm` with `pnpm`/`yarn` as needed.)

Again, after `wrangler deploy` succeeds:

- Your **Worker** is updated (control plane).
- Your private Worker-managed JSON stays as-is until you call the webhook rebuild endpoints.

This separation is intentional: you can adjust Worker logic or configuration without touching the stored JSON until you actually want to rebuild it.

### d) Always trigger `/build` after changes to route source

Whenever you change:

- Files under `STATIK_SRC` (e.g., `src-api/users/[id].js`), or
- Any logic that affects JSON output,

you should:

1. Commit and push (so Cloudflare redeploys the Worker via Git or `wrangler deploy`).
2. Then **trigger the build**:

   ```bash
   curl -X POST "https://api.example.com/build" \
     -H "Authorization: Bearer YOUR_STATIK_BUILD_TOKEN"
   ```

   or for a specific route:

   ```bash
   curl -X POST "https://api.example.com/build?route=/users/1" \
     -H "Authorization: Bearer YOUR_STATIK_BUILD_TOKEN"
   ```

Only after this POST completes will your R2-backed JSON be updated.

## 9) When to use the Cloudflare adapter

Use the **Cloudflare Worker + R2 adapter** when:

- You want **StatikAPI’s file-based DX**, but:
  - You don’t want to manage separate static hosting, or
  - You need a first-class **webhook endpoint** to trigger rebuilds.
- You’re already on Cloudflare and want:
  - **R2** for storage
  - **Workers** for routing, manifests, and build webhooks
  - KV for a tiny manifest

For pure static hosting of `api-out/` only (no Worker control plane), see:
